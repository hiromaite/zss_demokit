# System Requirements Draft

## 1. 文書の目的

本書は、新しいセンサ計測・可視化システムの要件定義の叩き台である。

対象は以下の 2 種類のデバイスを、単一の Python GUI アプリケーションから
個別に接続・制御・可視化するシステムである。

- BLE 接続のワイヤレスセンサデバイス
- シリアル接続のワイヤードセンサデバイス

両デバイスは同時接続しない前提とする。

## 2. 背景

現行資産では、以下のような分断がある。

- 旧 BLE 系は `old_firmware` と `old_gui` によって構成されている
- 参考 GUI は `example_gui` にあり、Python / PySide6 / pyqtgraph を採用している
- 通信方式が統一されていない
  - BLE 系: 固定長バイナリ通知
  - wired 系: テキスト行ベース
- GUI の見た目と操作感に一貫性がない
- old_firmware は 50 ms 周期の安定性に課題がある

## 3. システムスコープ

### 3.1 対象

- Windows 向けに配布可能な Python GUI アプリ
- GUI から接続される 2 種類のセンサデバイス
- センサデータの取得、表示、記録、簡易制御
- `PyInstaller` による Windows 11 Pro 向けパッケージ
- 管理者権限なしで実行できる配布形態

### 3.2 対象外

- 2 台同時接続
- クラウド連携
- 複数 PC 間でのデータ共有
- スマートフォン向けアプリ
- v1 における校正機能
- 旧資産との後方互換の維持

### 3.3 前提条件

- Wired デバイスのハードウェア構成は BLE デバイスと同系統を想定する
- Wired firmware は BLE firmware の派生または再設計を許容する
- Wired デバイスは v1 では `Windows COM port` として扱う
- Wired の default serial setting は `115200 baud`, `8N1` とする
- GUI の日常開発、試作確認、初期テストは macOS ローカル環境で行う
- Python 実行環境は local prototype 段階から `Python 3.12` を使う

## 4. 利用シナリオ

### 4.0 起動時 / モード切替

- GUI 起動時には、スプラッシュ / ランチャー画面で `BLE mode` または `Wired mode` を選択する
- 起動後も設定画面または設定モーダルからモード変更できる
- モード変更時に同時接続状態は作らない

### 4.1 BLE デバイス利用時

- ユーザーは GUI から BLE デバイスを検索して接続する
- GUI は 50 から 100 ms 程度のサンプリングで受信したデータを表示する
- GUI は最低限 `Pump ON/OFF` と `Get Status` を行える
- GUI はセッション記録を保存できる

### 4.2 wired デバイス利用時

- ユーザーは GUI から COM ポートを選択して接続する
- GUI は v1 では `115200 baud` を既定値として使う
- GUI は 10 ms 周期を必須要件とする高頻度データを表示する
- GUI は BLE と同一の論理機能として `Pump ON/OFF` と `Get Status` を行える
- GUI はセッション記録を保存できる

## 5. 機能要求

### 5.1 共通要求

- 単一の GUI アプリで BLE デバイスと wired デバイスの両方を扱えること
- 起動時に動作モードを明示的に選択できること
- 起動後に設定画面または設定モーダルから動作モードを変更できること
- 接続対象に応じて UI と内部処理を切り替えられること
- 同時接続は不可とし、常にアクティブデバイスは 1 台であること
- リアルタイムでセンサ値を可視化できること
- セッションデータをファイル保存できること
- 接続、切断、エラー、記録状態を明確に表示できること
- デバイス能力情報やバージョン情報を取得できること
- BLE / wired で共通の記録フォーマットを採用すること
- UI の表示言語は英語を基本とすること

### 5.2 GUI 要求

- `example_gui` に近い、業務用途向けで落ち着いた見た目を目指すこと
- `old_gui` のような単一 HTML への密結合構成は避けること
- レイアウトは接続、状態、制御、グラフ、ログが整理された構成であること
- 起動時の launcher は compact で、過度な占有面積を持たないこと
- グラフの操作性は `example_gui` と同等の方向性を維持すること
- オートスケールとマニュアルスケールを明確に切り替えられること
- 時間軸スパンの切り替え、ズーム、パン、リセットが容易であること
- 長時間運用でも操作性が落ちにくいこと
- UI 文言は v1 から英語表記とすること
- 状態表示は重複した top-level bar を必須にせず、主要パネルに整理して配置できること
- 設定導線は左側の制御領域に自然に統合できること
- 記録状態や file path などの dynamic text により panel 幅が揺れないこと

### 5.3 BLE デバイス向け要求

- 接続、切断、再接続が GUI から安定して行えること
- センサデータ通知は 50 から 100 ms 程度を目標値として扱えること
- `Pump ON/OFF` と `Get Status` をサポートすること
- 可能な限り既存の BLE device name / UUID を維持すること
- BLE 通信のばらつきと切断を GUI 側で扱えること

### 5.4 wired デバイス向け要求

- ポート選択、接続、切断が GUI から簡単に行えること
- 10 ms 周期の更新を必須要件として表示・記録できること
- BLE と同一の論理機能として `Pump ON/OFF` と `Get Status` をサポートすること
- デバイス能力情報を取得できること
- 高頻度受信時も GUI の描画が破綻しにくいこと

### 5.5 表示要求

両モードで共通の主要表示項目は以下とする。

- `Zirconia Output Voltage` [V]
- `Heater RTD Resistance` [Ohm]
- `Flow Rate` [L/min]

### 5.6 記録要求

- セッションごとにファイルを作成できること
- 記録中断時に partial file を残し、次回起動時に検出できること
- メタデータをヘッダとして保存できること
- BLE / wired で共通の記録フォーマットを採用すること
- 記録時刻は GUI 受信時刻を基準とすること
- 将来の解析に必要な時刻、デバイス種別、状態、セグメント情報を残せること

### 5.7 v1 で対象外とする機能

- 校正機能
- 旧資産との後方互換
- 自動再接続

## 6. 非機能要求

### 6.1 性能

- BLE モード
  - 50 から 100 ms 程度の受信を目標値として安定して描画・記録できること
- wired モード
  - 10 ms 周期の受信を必須要件として扱い、表示遅延を抑えられること
- グラフ更新と記録処理は、受信スレッドを不必要にブロックしないこと
- 高頻度な wired 通信では、通信安定性を最優先として transport を選定すること

### 6.2 保守性

- transport 差分を GUI の主要画面実装から分離すること
- 通信仕様はドキュメント化され、バージョン管理されること
- GUI は単一巨大ファイルに寄せず、責務ごとに分割すること
- GUI レイアウトの主要比率は content length に依存しない構成にすること

### 6.3 配布性

- Windows 向けにパッケージ化できること
- `PyInstaller` を主経路とすること
- 対象 OS は `Windows 11 Pro` とすること
- インストーラなしで配布できること
- 管理者権限なしで運用できること
- 初回セットアップの負担が低いこと

### 6.4 信頼性

- 切断時や異常時に GUI がハングしにくいこと
- データ保存中のクラッシュ時に完全消失しにくいこと
- デバイス応答異常や capability 不一致を UI 上で明示できること
- 異常時の基本動作は警告表示とし、v1 では自動再接続を必須にしないこと

### 6.5 開発優先度

優先順位は以下の順とする。

1. 通信安定性
2. UI / UX
3. 将来拡張性

## 7. 旧システムから引き継ぐ資産

### 7.1 引き継ぎ候補

- old_firmware のセンサ取得ロジック
- old_firmware の状態フラグ設計
- old_firmware の BLE 制御機能
- example_gui の GUI 構成方針
- example_gui の記録管理、partial file 検出、設定保存
- example_gui の pyqtgraph を使ったプロット UI

### 7.2 そのままは引き継がない候補

- old_gui の単一 HTML / JS 構成
- transport ごとにばらばらなメッセージモデル
- old_firmware の周期ゆらぎを許容する前提

## 8. 設計上の主要課題

- 2 種類の transport を 1 つの GUI アプリでどう抽象化するか
- 起動時モード選択と実行中モード切替をどう一貫した UX にするか
- GUI 内部のデータモデルを transport 非依存にできるか
- BLE と wired で同一プロトコルをどこまで共有するか
- 高頻度データ受信時の描画負荷をどう制御するか
- wired の adapter / parser を GUI 本体からどう分離するか

## 9. 初期方針

- GUI は Python / PySide6 を第一候補とする
- グラフは pyqtgraph を第一候補とする
- GUI 内部には transport 非依存の共通データモデルを置く
- 通信仕様は「共通の論理メッセージ」と「transport 別の符号化」を分けて考える
- 新 firmware では「計測周期の安定化」を優先課題に置く
- v1 の UI は英語表記とする
- BLE / wired で共通の記録フォーマットを採用する
- 表示寄りの派生値は GUI 側計算を第一候補とする
- 校正機能と後方互換は v1 の対象外とする
- canonical field 名と command 名は `protocol_catalog_v1.md` を基準とする
- 記録ファイルスキーマは `recording_schema.md` を基準とする

## 10. 受け入れ観点の叩き台

- 起動時に `BLE mode` / `Wired mode` を選択できる
- 設定画面または設定モーダルからモード変更に入れる
- BLE デバイス接続時に、接続、計測、制御、記録が一通り動作する
- wired デバイス接続時に、接続、計測、制御、記録が一通り動作する
- 両モードで `Zirconia Output Voltage`、`Heater RTD Resistance`、`Flow Rate` を表示できる
- GUI の見た目とレイアウトに統一感がある
- compact launcher が過度に大きくない
- 状態表示の重複した top bar がなくても操作に必要な情報へ到達できる
- recording 開始や status 文言変更後も panel 幅が視覚的に崩れない
- マニュアルスケール操作が実用的である
- 通信異常時に状態がわかりやすい
- 記録ファイルが後から再利用しやすく、両モードで共通スキーマである

## 11. Open Questions

- Wired firmware を BLE firmware の派生として構成するか、共通モジュール化した新設計にするか
- BLE の既存 UUID を維持しつつ capability / status 拡張をどう載せるか
- raw payload の debug 保存を v1 で持つかどうか
